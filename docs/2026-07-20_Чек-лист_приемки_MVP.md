# Чек-лист приёмки MVP

Этот документ нужен для быстрой проверки, что MVP реально готов к работе после обновлений, правок или выкладки на сервер.

## 1. Что должно быть запущено

- Django-админка открывается по адресу `/admin/`
- `/health/` отвечает `{"status": "ok"}`
- Telegram-бот отвечает на `/start`
- включены сервисы:
  - `referral_admin`
  - `referral_bot`
  - `nginx`
- включены таймеры:
  - `bonus_expiration_warning.timer`
  - `bonus_expiration.timer`

## 2. Быстрая техническая проверка на сервере

Выполнить:

```bash
cd /opt/sistema_loialnosti/backend
source .venv/bin/activate
python manage.py migrate
python manage.py collectstatic --noinput
systemctl restart referral_admin
systemctl restart referral_bot
systemctl restart nginx
systemctl status referral_admin --no-pager
systemctl status referral_bot --no-pager
systemctl status nginx --no-pager
systemctl status bonus_expiration_warning.timer --no-pager
systemctl status bonus_expiration.timer --no-pager
```

Ожидаемый результат:

- миграции применяются без ошибок
- `collectstatic` завершается без ошибок
- сервисы в статусе `active (running)`
- таймеры в статусе `active (waiting)`

## 3. Проверка админки

Проверить в браузере:

- открывается страница входа в админку
- после входа открывается главный экран админки
- на главном экране есть:
  - приоритеты на день
  - быстрые переходы
  - сводка по заявкам и списаниям

Проверить разделы:

- `Реферальные заявки`
- `Участники`
- `Списания бонусов`
- `Операции по бонусам`

Что должно работать:

- русские названия колонок и фильтров
- визуальные статусы
- быстрые действия из списков
- карточки с памяткой администратора

## 4. Проверка регистрации участника в Telegram

Нужно проверить на новом тестовом пользователе:

1. Отправить `/start`
2. Пройти регистрацию
3. Указать имя
4. Указать телефон
5. Подтвердить согласие

Ожидаемый результат:

- бот завершает регистрацию
- пользователь попадает в основное меню
- в админке появляется новый участник

## 5. Проверка реферальной ссылки

Проверить:

1. Участник получает персональную ссылку
2. Участник видит готовый текст для отправки
3. По ссылке открывается бот
4. Новый пользователь может оставить заявку

Ожидаемый результат:

- заявка сохраняется в админке
- администратору приходит уведомление в Telegram
- в заявке видно, кто пригласил клиента

## 6. Проверка собственной заявки участника

Проверить:

1. Зарегистрированный участник оставляет заявку для своей компании
2. Указывает продукцию, тираж и комментарий

Ожидаемый результат:

- заявка создаётся
- в админке видно, что это своя заявка, а не реферал
- администратору приходит уведомление

## 7. Проверка подарков и списания бонусов

Проверить:

1. Участник открывает раздел подарков
2. Выбирает подарок
3. Подтверждает отправку запроса

Ожидаемый результат:

- создаётся запрос на списание бонусов
- администратору приходит Telegram-уведомление
- в уведомлении и в админке есть:
  - имя участника
  - телефон
  - подарок или комментарий
  - сумма списания

## 8. Проверка начисления и подтверждения бонусов

В админке проверить вручную:

1. Открыть заявку клиента
2. Перевести её по статусам
3. Подтвердить начисление бонуса

Ожидаемый результат:

- у участника меняется баланс
- запись появляется в истории бонусов
- в боте участник видит новый баланс

## 9. Проверка аннулирования и минусового баланса

Проверить вручную:

1. У участника должно быть подтверждённое начисление
2. При необходимости часть бонусов должна быть уже потрачена
3. В админке создать аннулирование или ручную корректировку

Ожидаемый результат:

- баланс пересчитывается корректно
- если бонусы уже были потрачены, баланс может уйти в минус
- минусовой баланс отображается в админке и не ломает интерфейс

## 10. Проверка основного контакта компании

Проверить:

1. У нескольких участников одна и та же компания
2. Один участник назначен основным контактом
3. При необходимости основной контакт меняется

Ожидаемый результат:

- в компании можно явно определить основного контакта
- бонус по своей компании относится к основному контакту

## 11. Проверка предупреждения о сгорании бонусов

Проверить:

1. У начисления есть дата окончания
2. Запустить команду вручную:

```bash
python manage.py send_bonus_expiration_warnings
```

Ожидаемый результат:

- участник получает Telegram-предупреждение
- повторно одно и то же предупреждение не отправляется

## 12. Проверка сгорания бонусов

Проверить:

1. Есть начисление с прошедшей датой окончания
2. Запустить команду:

```bash
python manage.py expire_bonus_entries
```

Ожидаемый результат:

- сгорает только неиспользованный остаток
- в истории бонусов появляется отдельная операция сгорания
- повторно одно и то же начисление не сгорает второй раз

## 13. Что считается нормой для этапа 1

Для текущего MVP нормально, если:

- amoCRM ещё не подключён
- все спорные решения принимает администратор вручную
- бонусы подтверждаются вручную
- списания подтверждаются вручную
- участник не может сам “списать” бонусы без решения администратора

## 14. Когда MVP можно считать готовым к рабочему использованию

MVP можно считать готовым, если одновременно выполняется всё ниже:

- админка открывается
- бот отвечает
- участник регистрируется
- ссылка участника работает
- новая заявка доходит до админки
- подарок или списание доходит до админки
- Telegram-уведомления администратору приходят
- баланс пересчитывается корректно
- минусовой баланс не ломает систему
- предупреждение о сгорании и само сгорание работают

Если хотя бы один пункт выше не выполняется, MVP ещё нельзя считать полностью готовым к стабильной ручной работе.
